OVERVIEW
--------
All tests are run in MazeRunner_tb.sv using RemoteComm to send 16-bit commands
and RunnerPhysics to model robot motion, maze walls, gyro behavior, and IR/A2D
sensor responses. FAST_SIM=1 is enabled in the MazeRunner top level to keep
simulation runtime practical. Tests are self-checking and stop immediately on
failure with a descriptive message.

Tests are ordered and stateful: each one assumes the DUT state left by earlier
tests. The sequence covers reset behavior, sensor bring-up, command decoding,
heading control, movement, invalid-command behavior, and solver-mode entry.


TEST 1: PWM Midrail Check
-------------------------
What it does:
After reset release and short settling time, samples iDUT.iMTR.lft_duty and
iDUT.iMTR.rght_duty.

What it checks:
Both duties are near midrail (0x7E0 to 0x820). This confirms neutral motor
drive at startup and no unintended movement.

Signals checked:
iDUT.iMTR.lft_duty, iDUT.iMTR.rght_duty


TEST 2: NEMO Setup
------------------
What it does:
Waits for iPHYS.iNEMO.NEMO_setup to assert.

What it checks:
The inertial interface SPI initialization sequence completed before control
tests proceed.

Signals checked:
iPHYS.iNEMO.NEMO_setup


TEST 3: Calibrate Command (0x0000)
----------------------------------
What it does:
Sends command 0x0000 over UART via RemoteComm, waits for cal_done, then waits
for response byte.

What it checks:
End-to-end command path and calibration flow:
RemoteComm -> UART_wrapper -> cmd_proc -> strt_cal -> inert_intf ->
inertial_integrator -> cal_done -> cmd_proc RESP -> UART_wrapper TX.
Response must be 0xA5.

Signals checked:
iDUT.cal_done, resp_rdy, resp


TEST 4: Heading Change West (0x23FF)
------------------------------------
What it does:
Sends heading command 0x23FF (west), lets control settle, then inspects wheel
speed direction and movement state before waiting for ACK.

What it checks:
Heading command decode and turn behavior:
- navigate enters HEADING and asserts moving
- PID produces CCW turn direction (lft_spd < 0 and rght_spd > 0)
- command completes and responds with 0xA5

Signals checked:
iDUT.moving, iDUT.lft_spd, iDUT.rght_spd, iDUT.iMTR.*duty, resp


TEST 5: Heading Change South (0x27FF)
-------------------------------------
What it does:
Sends heading command 0x27FF (south), waits for ACK, checks heading closeness,
and waits for robot angular velocities to settle before the next move test.

What it checks:
Back-to-back heading commands are accepted and completed correctly. Also ensures
robot is settled before forward movement test to avoid transient physics errors.

Signals checked:
iDUT.actl_hdng, resp, iPHYS.omega_lft, iPHYS.omega_rght


TEST 6: Move Forward (0x4000)
-----------------------------
What it does:
Sends move command 0x4000 and waits until RunnerPhysics reports forward wheel
motion (omega_sum exceeds threshold), then checks ACK and command-mode return.

What it checks:
MOVE command decode and acceleration path:
- cmd_proc asserts strt_mv
- navigate ramps frwrd_spd
- PID + MtrDrv generate forward wheel drive
- command returns to cmd mode after completion

Signals checked:
iPHYS.omega_sum, resp, iDUT.cmd_md


TEST 7: Invalid Opcode Has No Response (0xE000)
-----------------------------------------------
What it does:
Sends unsupported opcode 0xE000 and watches a bounded window for response.

What it checks:
Invalid commands do not generate spurious responses.

Signals checked:
resp_rdy edge activity, resp


TEST 8: Move Stop-Bit Decode + Response (0x4003)
------------------------------------------------
What it does:
Sends MOVE command 0x4003 (both stop bits set), waits a few cycles, checks
latched stop bits in cmd_proc, then checks ACK and cmd-mode return.

What it checks:
Correct decoding and latching of cmd[1:0] for MOVE:
- stp_lft = 1
- stp_rght = 1
Also confirms response and control-flow completion.

Signals checked:
iDUT.iCMD.stp_lft, iDUT.iCMD.stp_rght, resp, iDUT.cmd_md


TEST 9: Solve Command Enters Solver Mode (0x6001)
-------------------------------------------------
What it does:
Sends solve command 0x6001 and waits for cmd_md to go low.

What it checks:
cmd_proc transitions to solver mode and hands control to maze_solve path.

Signals checked:
iDUT.cmd_md


================================================================================
TIMEOUT POLICY
Each wait task has a bounded cycle limit (typically 50k to 2M cycles depending
on subsystem latency). If a condition is not met in time, the test prints FAIL
with context and halts via $stop.

A global 100 ms simulation timeout (#100_000_000) is also active to prevent
hung runs in case a local timeout is missed.
